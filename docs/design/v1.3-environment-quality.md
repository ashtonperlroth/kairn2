## 1. Structured CLAUDE.md Template

### Problem
The LLM currently freestyles CLAUDE.md content. Quality varies between runs. Sometimes generic, sometimes too long.

### Solution
Add a mandatory template to the compilation prompt:

```
# {Project Name}

## Purpose
{one-line description}

## Tech Stack
{bullet list of frameworks/languages}

## Commands
{concrete build/test/lint/dev commands}

## Architecture
{brief folder structure, max 10 lines}

## Conventions
{3-5 specific coding rules}

## Key Commands
{list /project: commands with descriptions}

## Output
{where results go, key files}
```

### Implementation
- Update `src/compiler/prompt.ts` to include the template
- Add "CLAUDE.md must follow this exact structure" instruction
- Add "Maximum 100 lines" constraint

## 2. Shell-Integrated Commands

### Problem
Current slash commands are static instructions. They don't include live project data.

### Solution
Use Claude Code's `!` syntax to embed shell output:

```markdown
# .claude/commands/review.md
Review the staged changes for quality and security:

!git diff --staged

Run the tests and check for failures:

!npm test 2>&1 | tail -20

Focus on: security, error handling, test coverage.
```

```markdown
# .claude/commands/status.md
Show current project state:

!git status --short
!git log --oneline -5

Read docs/TODO.md and summarize progress.
```

### Implementation
- Update compilation prompt to instruct LLM: "Commands that reference code should use ! for live output"
- Add `/project:status` to the "always include" list
- Add `/project:fix` with `$ARGUMENTS` for issue numbers
- Update adapter to handle `!` syntax (no changes needed — it's just markdown)

## 3. Path-Scoped Rules

### Problem
Current rules are unconditional (always loaded). For larger projects, this wastes context on irrelevant instructions.

### Solution
Generate path-scoped rules for code projects:

```markdown
# .claude/rules/api.md
---
paths:
  - "src/api/**"
  - "src/routes/**"
---
- All handlers return { data, error } shape
- Use Zod for request validation
- Log errors with request ID context
- Never expose internal error details to clients
```

```markdown
# .claude/rules/testing.md
---
paths:
  - "tests/**"
  - "**/*.test.*"
  - "**/*.spec.*"
---
- Use AAA pattern: Arrange-Act-Assert
- One assertion per test when possible
- Test names describe behavior: "should_return_empty_when_no_items"
- Mock external dependencies, never real APIs
```

### Implementation
- Update compilation prompt with path-scoped rule examples
- Scanner should detect `src/api`, `src/routes`, `tests/`, `__tests__/` to decide which scoped rules to generate
- Keep `security.md` and `continuity.md` as unconditional

## 4. Hooks in settings.json

### Problem
Generated settings.json only has permissions. No automation hooks.

### Solution
Generate hooks based on project type:

**All projects:**
```json
{
  "hooks": {
    "PreToolUse": [{
      "matcher": "Bash",
      "hooks": [{
        "type": "command",
        "command": "CMD=$(cat | jq -r '.tool_input.command // empty') && echo \"$CMD\" | grep -qiE 'rm\\s+-rf\\s+/|DROP\\s+TABLE|curl.*\\|\\s*sh' && echo 'Blocked destructive command' >&2 && exit 2 || true"
      }]
    }]
  }
}
```

**Code projects with Prettier:**
```json
{
  "hooks": {
    "PostToolUse": [{
      "matcher": "Edit|Write",
      "hooks": [{
        "type": "command",
        "command": "FILE=$(cat | jq -r '.tool_input.file_path // empty') && [ -n \"$FILE\" ] && npx prettier --write \"$FILE\" 2>/dev/null || true"
      }]
    }]
  }
}
```

### Implementation
- Update `src/adapter/claude-code.ts` to merge hooks into settings.json
- Update compilation prompt to instruct LLM on when to include which hooks
- Scanner should detect Prettier/ESLint/Black to choose the right formatter hook

## 5. New Slash Commands

### /project:status
```markdown
Show current project state:

!git status --short
!git log --oneline -5
!cat docs/TODO.md 2>/dev/null || echo "No TODO.md yet"

Summarize: what changed recently, what's in progress, what's next.
```

### /project:fix
```markdown
Fix issue #$ARGUMENTS

1. Read the issue description (use GitHub MCP if available, or ask user to paste it)
2. Plan the fix in docs/DECISIONS.md
3. Implement the fix
4. Write tests for the fix
5. Run the test suite to verify
6. Create a commit: "fix: description (closes #$ARGUMENTS)"
```

### Implementation
- Add to "always include for code projects" list in compilation prompt
- These are static templates — the LLM doesn't need to generate them

## 7. Improved TDD Skill

### Current
Generic "write tests first" instruction.

### Improved (3-agent isolation pattern from alexop.dev)
```markdown
---
name: tdd-workflow
description: Test-Driven Development with isolated phases
---

# TDD Workflow

## Mandatory Cycle
1. **RED:** Write a failing test for the feature. Do NOT write implementation.
   Verify the test FAILS before proceeding.
2. **GREEN:** Write the MINIMUM code to make the test pass.
   Do NOT add features beyond what the test requires.
3. **REFACTOR:** Improve code quality while keeping tests green.
   Focus on: extract functions, reduce duplication, clarify names.

## Rules
- Never write tests and implementation in the same step
- Test names describe behavior, not implementation
- Use AAA pattern: Arrange-Act-Assert
- One assertion per test when possible
- Run the test suite after each phase to verify state
```

### Implementation
- Replace current generic TDD skill in compilation prompt examples
- Add as a static skill template that the LLM can reference
