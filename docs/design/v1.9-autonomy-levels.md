# v1.9.0 Design Doc — Autonomy Levels & Workflow Loops

> Let users choose how autonomous their environment is during setup. From guided orientation to fully autonomous development loops with a PM agent.

**Status:** Planned  
**Depends on:** v1.8.0

---

## The Core Idea

During `kairn describe` or `kairn init`, the user chooses an autonomy level. This determines what gets generated — not just which files, but the entire workflow philosophy:

```
$ kairn describe "Build a Next.js app with Supabase auth"

  ...clarification questions...

  Autonomy level?
  › 1. Guided       — orientation + commands, you drive everything
    2. Assisted     — workflow loop, you approve each phase
    3. Autonomous   — PM agent plans, loop executes, you review PRs
    4. Full Auto    — continuous execution (⚠ like --dangerously-skip-permissions)
```

Each level generates a progressively more autonomous environment.

---

## Level 1: Guided (Default)

**Philosophy:** "I'll show you the tools and you decide when to use them."

### What Gets Generated
- Everything from current Kairn output (CLAUDE.md, commands, rules, skills, agents, MCP servers)
- **`/project:tour`** — interactive walkthrough of the environment
- **SessionStart welcome** — brief orientation on first run
- **Workflow section in CLAUDE.md** — "When to use what" reference table
- **`QUICKSTART.md`** — human-readable guide (not for Claude, for the user to read in an editor)

### /project:tour Command
```markdown
# Environment Tour

Welcome! Let me show you around this Kairn environment.

## Your Commands
Read .claude/commands/ and list each one with a one-line description.
Group them by workflow phase:

  PLAN:    /project:spec, /project:sprint, /project:plan
  BUILD:   (just start coding — Claude reads CLAUDE.md automatically)
  VERIFY:  /project:prove, /project:grill, /project:test
  SHIP:    /project:commit, /project:review
  MANAGE:  /project:status, /project:tasks, /project:reset

## Your Agents
Read .claude/agents/ and explain each one:
  @reviewer  — invoke with "@reviewer check my changes"
  @linter    — invoke with "@linter run static analysis"

## Your MCP Tools
!claude mcp list 2>/dev/null || echo "Run /mcp in Claude Code to see active tools"

## Workflow
For this project, the recommended flow is:
  spec → sprint → plan → code → prove → grill → commit

## Tips
- Type / to see all commands
- Type @ to invoke an agent
- Paste raw errors — don't summarize them
- Use subagents for deep investigation
- Say "update CLAUDE.md" after any correction

Ready to start? Try /project:spec to define your first feature.
```

## Level 2: Assisted (Workflow Loop)

**Philosophy:** "I'll run the loop, you approve each phase."

### What Gets Generated
- Everything from Level 1
- **`/project:loop`** — workflow-specific automated loop
- **`@pm` agent** — planning and spec agent

### /project:loop Command (Code Projects)
```markdown
# Development Loop

Run an assisted development cycle for the next feature.

## Phase 1: SPEC
Use @pm to review docs/TODO.md and docs/SPRINT.md.
If no sprint is defined, run /project:spec to interview the user.
Wait for user approval of the spec.

## Phase 2: PLAN
Read the approved spec in docs/SPRINT.md.
Plan the implementation: files to change, tests to write, approach.
Write plan to docs/DECISIONS.md.
Wait for user approval of the plan.

## Phase 3: IMPLEMENT
Follow the plan. Implement the feature.
Run tests after each change.
Commit each logical unit: "feat: description"

## Phase 4: VERIFY
Run /project:prove to verify the implementation.
If confidence is LOW or MEDIUM, fix issues and re-verify.

## Phase 5: REVIEW
Run /project:grill for adversarial review.
Fix any BLOCKERs.

## Phase 6: SHIP
Run /project:commit.
Report what was built and what's next from docs/TODO.md.

Then ask: "Continue to next feature?"
If yes, return to Phase 1.
```

### /project:loop Command (Research Projects)
```markdown
# Research Loop

Run an assisted research cycle.

## Phase 1: QUESTION
Review docs/TODO.md for the next research question.
If none, ask the user what to investigate.

## Phase 2: RESEARCH
Use @researcher to search, extract, and analyze sources.
Log findings to docs/SOURCES.md and docs/LEARNINGS.md.

## Phase 3: SYNTHESIZE
Review all findings. Write structured summary to docs/SUMMARY.md.
Cite all sources.

## Phase 4: REVIEW
Present the summary. Ask the user for feedback.
Revise based on feedback.

## Phase 5: NEXT
Update docs/TODO.md — mark question as done, identify follow-ups.
Ask: "Continue to next question?"
```

### @pm Agent
```markdown
---
name: pm
description: Project manager agent. Maintains roadmap, specs features, prioritizes work.
tools: Read, Write(docs/**), Glob, Grep
model: opus
permissionMode: default
memory: project
---

You are a project manager for this codebase.

Your responsibilities:
1. Maintain docs/TODO.md — keep it prioritized and current
2. Write specs to docs/SPRINT.md when asked
3. Review completed work and suggest what's next
4. Track decisions in docs/DECISIONS.md
5. Track learnings in docs/LEARNINGS.md

When invoked:
- Read all docs/ files to understand current state
- Read recent git log for what changed
- Suggest the highest-priority next task
- If asked to spec something, interview the user (5-8 questions)

You do NOT write code. You plan, spec, and prioritize.
```

## Level 3: Autonomous (PM + Loop)

**Philosophy:** "The PM plans, the loop executes, you review PRs."

### What Gets Generated
- Everything from Level 2
- **`/project:auto`** — PM-driven autonomous loop
- **Worktree isolation** — all work happens in isolated branches
- **PR-based delivery** — every feature is a PR for review

### /project:auto Command
```markdown
# Autonomous Development

PM-driven development loop with PR delivery.

## Phase 1: PLAN (@pm)
Use @pm to:
- Read docs/TODO.md and docs/SPRINT.md
- Select the highest-priority unfinished task
- Write a spec to docs/SPRINT.md
- Present the spec for approval

Wait for user approval. If approved, proceed.

## Phase 2: BRANCH
Create an isolated worktree:
  git worktree add ../project-feat-{name} -b feat/{name}
All implementation happens in the worktree.

## Phase 3: IMPLEMENT (@implementer)
Delegate to @implementer in the worktree:
- Follow the spec in docs/SPRINT.md
- Build, test, commit after each change

## Phase 4: VERIFY (@qa-orchestrator)
Delegate to @qa-orchestrator:
- @linter runs static analysis
- Run functional tests
- @e2e-tester runs browser tests (if available)
If NEEDS FIXES: @implementer fixes, re-verify.

## Phase 5: PR
Create a pull request:
  gh pr create --title "feat: {name}" --body "{spec + QA report}"

## Phase 6: NEXT
Report:
  "PR #{N} ready for review: {link}
   Next priority from TODO.md: {next task}
   Continue? (y/n)"

If yes, return to Phase 1 with next task.
```

## Level 4: Full Auto (⚠ Advanced)

**Philosophy:** "Run continuously. I'll review when I'm ready."

### What Gets Generated
- Everything from Level 3
- **`/project:autopilot`** — continuous execution without approval gates
- **Warning in CLAUDE.md** about autonomous operation
- **Stop conditions** (max features, max cost, time limit)

### /project:autopilot Command
```markdown
# ⚠ Autopilot Mode

Continuous autonomous development. The PM plans, the loop executes,
PRs are opened automatically. You review when ready.

## Configuration
- Max features per session: 5 (prevent runaway)
- Stop on: test failure, build error, or blocked dependency
- All work in isolated worktrees
- Every feature = one PR

## The Loop
Repeat until max features reached or stopped:
1. @pm selects next priority from docs/TODO.md
2. Create worktree + branch
3. @implementer builds the feature
4. @qa-orchestrator verifies
5. Open PR via gh
6. Report status
7. Move to next feature

## Stop Conditions
- Max 5 features per autopilot session
- Any BLOCKER from @qa-orchestrator
- Build failure that @implementer can't resolve in 3 attempts
- User presses Escape

## ⚠ Warning
This mode operates with minimal human oversight.
Recommended only for projects with strong test suites.
Review all PRs before merging.
```

### CLAUDE.md Warning (Level 4 only)
```markdown
## ⚠ Autopilot Mode Active
This environment is configured for autonomous operation.
The @pm agent plans features and /project:autopilot executes them.
All changes are delivered as PRs — review before merging.
Stop conditions: max 5 features, test failure, or Escape.
```

## Implementation

### 1. Add Autonomy Level to Clarification Flow
In `src/commands/describe.ts`, after the clarification questions, add:

```typescript
const autonomyLevel = await select({
  message: "Autonomy level",
  choices: [
    { name: "1. Guided — orientation + commands, you drive", value: 1 },
    { name: "2. Assisted — workflow loop, you approve phases", value: 2 },
    { name: "3. Autonomous — PM plans, loop executes, you review PRs", value: 3 },
    { name: "4. Full Auto — continuous execution (⚠ advanced)", value: 4 },
  ],
  default: 1,
});
```

### 2. Pass Autonomy Level to Compiler
Add `autonomy_level` to the enriched intent passed to the LLM:

```
Autonomy level: 2 (Assisted)
- Generate /project:loop command for the workflow type
- Generate @pm agent
- Include workflow flowchart in CLAUDE.md
```

```typescript
export interface EnvironmentSpec {
  // ... existing fields
  autonomy_level: 1 | 2 | 3 | 4;
}
```
