## 1. Sprint Contract Pattern

### Background (from Anthropic)
Before coding a feature, the Generator and Evaluator negotiate a "sprint contract" — an agreement on what "done" looks like and how it will be tested. This bridges user stories to verifiable implementation.

### `/project:sprint` Command
```markdown
# .claude/commands/sprint.md
Define a sprint contract for the next feature:

1. What feature are we building? (if not specified: read docs/TODO.md)
2. Write a CONTRACT to docs/SPRINT.md:
   - Feature name
   - Acceptance criteria (specific, testable)
   - How to verify (manual steps or test commands)
   - What files will be created/modified
   - Estimated scope (small/medium/large)
3. Do NOT start coding until the contract is confirmed.

!cat docs/TODO.md 2>/dev/null
```

### Implementation
- Add `/project:sprint` to code project command set
- Add `docs/SPRINT.md` to generated docs
- Update continuity rule to reference SPRINT.md

## 2. Multi-Agent QA Pipeline

### Background (from Anthropic)
Anthropic's harness research shows that separating evaluation into specialized agents dramatically improves quality. A dedicated QA agent using Playwright catches "stub" implementations. A linter agent catches code quality issues the implementer self-corrects poorly on.

### Architecture
```
@qa-orchestrator
├── @linter          (static analysis — haiku, fast, cheap)
│   ├── prettier/eslint/biome (JS/TS)
│   ├── black/ruff/mypy (Python)
│   └── semgrep (security)
├── build + typecheck (direct — npm run build, tsc --noEmit)
├── functional tests  (direct — run checklist items)
└── @e2e-tester      (browser automation — sonnet, Playwright)
    ├── navigate, click, fill, submit
    ├── verify behavior (not just DOM existence)
    └── screenshot failures
```

### Agent Definitions

**QA Orchestrator** — delegates to specialized agents, compiles report:
```markdown
---
name: qa-orchestrator
description: Orchestrates QA by delegating to linter, e2e-tester
tools: Read, Bash, Glob, Grep, Agent(linter, e2e-tester)
model: sonnet
permissionMode: plan
---
```

**Linter** — fast static analysis pass:
```markdown
---
name: linter
description: Runs formatters, linters, security scanners
tools: Read, Bash, Glob, Grep
model: haiku
permissionMode: plan
---
```

**E2E Tester** — browser-based QA:
```markdown
---
name: e2e-tester
description: Tests via Playwright browser automation
tools: Read, Bash, Glob
model: sonnet
mcpServers: ["playwright"]
---
```

### Implementation
- Add all three agents to `.claude/agents/`
- QA orchestrator uses `Agent(linter, e2e-tester)` in tools to spawn sub-agents
- E2E tester only included when Playwright is in the selected tools
- Linter detects available tools automatically (no hardcoded assumptions)
- `/project:ship` delegates Phase 4 to `@qa-orchestrator` instead of the old `@verifier`

## 3. PostCompact Context Re-injection

### Problem
When Claude's context window fills and it runs compaction, critical project rules and current task context can be lost.

### Solution
A PostCompact hook that re-injects essential context:

```json
{
  "hooks": {
    "PostCompact": [{
      "matcher": "",
      "hooks": [{
        "type": "prompt",
        "prompt": "Re-read CLAUDE.md and docs/SPRINT.md (if it exists) to restore project context after compaction."
      }]
    }]
  }
}
```

### Implementation
- Add PostCompact hook to generated settings.json for all projects
- Keep the re-injection minimal (just a pointer to read files, not inline content)

## 4. Context Budget Enforcement

### Problem
The LLM sometimes selects too many MCP servers or generates overly verbose CLAUDE.md files.

### Solution
Add explicit context budget to the compilation prompt:

```
## Context Budget (STRICT)
- MCP servers: maximum 6. Prefer fewer.
- CLAUDE.md: maximum 100 lines.
- Rules: maximum 5 files, each under 20 lines.
- Skills: maximum 3. Only include directly relevant ones.
- Agents: maximum 2. One reviewer + one specialist.
- Commands: no limit (loaded on demand, zero context cost).
- Hooks: maximum 4 (auto-format, block-destructive, protect-secrets, plus one contextual).

If the workflow doesn't clearly need a tool, DO NOT include it.
Each MCP server costs 500-2000 tokens of context window.
```

### Implementation
- Add budget section to `src/compiler/prompt.ts`
- Add post-compilation validation in `compile.ts`:
  - Warn if > 8 MCP servers
  - Warn if CLAUDE.md > 150 lines
  - Warn if > 5 skills

## 5. Optimize --diff

### Problem
`kairn optimize` overwrites all files. Users want to preview changes first.

### Solution
```bash
kairn optimize --diff
```

Shows a unified diff of what would change:
```
.claude/CLAUDE.md
  + ## Conventions
  + - Use Zod for validation
  - (no conventions section)

.claude/rules/testing.md
  + NEW FILE (path-scoped to tests/**)

.mcp.json
  + "sentry": { ... }
  - (no sentry server)

settings.json
  + hooks.PreToolUse (block destructive commands)
```

### Implementation
- Read existing files before writing
- Generate new spec
- Diff each file pair
- Display with chalk colors (green=add, red=remove)
- Prompt: "Apply these changes?"

## 6. Kairn Doctor

### Problem
Users modify generated environments manually and may break compatibility or miss best practices.

### Solution
```bash
kairn doctor
```

Validates the current .claude/ directory against best practices:

```
kairn doctor

  Checking .claude/ environment...

  ✅ CLAUDE.md exists (87 lines — good)
  ✅ settings.json exists with deny rules
  ⚠️  .mcp.json has 12 MCP servers (recommended: ≤8)
  ❌ Missing /project:help command
  ❌ Missing security rule (rules/security.md)
  ✅ continuity rule present
  ⚠️  No hooks configured in settings.json
  ✅ 2 agents defined (reviewer, researcher)
  ⚠️  CLAUDE.md missing "Commands" section
  ✅ .env not readable (deny rule present)

  Score: 7/10
  Run kairn optimize to fix issues.
```

### Implementation
- New file: `src/commands/doctor.ts`
- Reuse scanner logic from `src/scanner/scan.ts`
- Define a checklist of best practices (weighted scoring)
- Output colored pass/warn/fail with explanations
