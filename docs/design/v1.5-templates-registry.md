# v1.5.0 Design Doc — Templates & Registry

> Introduce a template gallery for common workflows and enhance tool registry management, while also adding support for the Hermes runtime.

**Status:** Planned  
**Depends on:** v1.4.0  
**Research:** ~/Projects/kairn-internal/RESEARCH-V2.md

---

## 1. Template Gallery

### Problem
Users currently start with a blank slate or rely on `kairn describe`s LLM for common workflows (Next.js, API, Research). A curated set of high-quality templates would provide immediate value and consistent best practices.

### Solution
Implement a template gallery where users can browse and activate pre-built `EnvironmentSpec`s.

### Implementation
- **Template Storage:** Store curated `EnvironmentSpec` JSON files in `~/.kairn/templates/`. These will be static, high-quality examples, potentially generated by Kairn itself and then hand-tuned.
- **`kairn templates` command:**
  - Lists available templates from `~/.kairn/templates/`.
  - Shows `name`, `description`, `tools`, `commands`, `rules`, `skills`, `agents` for each template.
  - Allows filtering by category or keywords.
- **`kairn activate <template_id>`:** Reuse the existing `activate` command logic, but allow it to also look in `~/.kairn/templates/` if the ID is not found in `~/.kairn/envs/`.

## 2. Registry Management

### Problem
The tool registry (`src/registry/tools.json`) is static and bundled. Users cannot add custom tools or manage their own registry entries.

### Solution
Allow users to add custom tool definitions and merge them with the bundled registry.

### Implementation
- **User Registry Storage:** Create `~/.kairn/user-registry.json` for user-defined tools.
- **`loadRegistry()` function update:** Modify `src/compiler/compile.ts` to load tools from `src/registry/tools.json` AND `~/.kairn/user-registry.json`, merging them (user-defined tools take precedence by ID).
- **`kairn registry list` command:** List all active tools (bundled + user-defined). Output rich details (categories, auth, env_vars, signup_url).
- **`kairn registry add` command:**
  - Interactively prompt user for `id`, `name`, `description`, `category`, `auth` (none/api_key/oauth/connection_string), `env_vars` (if api_key/connection_string), `signup_url`, `install` details.
  - Validate input against `RegistryTool` interface.
  - Save new tool to `~/.kairn/user-registry.json`.

## 3. Community Tool Submissions

### Problem
No clear path for users to contribute their valuable custom tool definitions back to the public Kairn registry.

### Solution
Define a process for community contributions, initially via GitHub PRs.

### Implementation
- **Documentation:** Add a `CONTRIBUTING.md` guide to the GitHub repo explaining how to submit new tool definitions via PR to `src/registry/tools.json`.
- **Review Process:** Establish a clear review process for submissions (e.g., must have `env_vars`, `signup_url`, `install` details, clear `best_for` tags).
- (Future: `kairn registry submit` for a hosted marketplace)

---

## 4. Hermes Runtime Adapter

### Problem
Kairn currently generates environments only for Claude Code. Users of Hermes (and eventually other agent runtimes) cannot benefit from Kairn's compilation.

### Solution
Add a new adapter for Hermes, converting `EnvironmentSpec` into Hermes' `config.yaml` and skill structure.

### Implementation
- **New File:** `src/adapter/hermes-agent.ts`.
- **Hermes `config.yaml` generation:** Translate `EnvironmentSpec.tools` into Hermes `mcp_servers` configuration.
  - Map Kairn `RegistryTool` `install.mcp_config` to Hermes `mcp_servers` entries.
- **Hermes Skill Generation:** Convert `EnvironmentSpec.harness.skills` and `EnvironmentSpec.harness.commands` into `~/.hermes/skills/` markdown files.
  - Hermes skills accept markdown files directly.
- **CLI Update:** Add `--runtime hermes` option to `kairn describe` and `kairn optimize` commands.
  - Update `src/types.ts` to include `HermesInstall` interface in `RegistryTool`.
- **Compilation Prompt Update:** Instruct the LLM on how to generate Hermes-specific `harness` content (e.g., mapping commands/rules/agents to Hermes skills and config).

## Testing This Release

1.  **Template Gallery:**
    - Create a few test `EnvironmentSpec` JSON files in `~/.kairn/templates/`.
    - Run `kairn templates` — verify listing and details.
    - Run `kairn activate <template_id>` — verify environment is written correctly.
2.  **Registry Management:**
    - Run `kairn registry list` — verify shows bundled tools.
    - Run `kairn registry add` — add a new dummy tool.
    - Run `kairn registry list` again — verify dummy tool appears.
    - Run `kairn describe` — verify LLM can use the dummy tool (if intent matches).
3.  **Hermes Adapter:**
    - Run `kairn describe "research ML papers" --runtime hermes --yes`.
    - Verify `~/.hermes/config.yaml` contains correct MCP server entries.
    - Verify `~/.hermes/skills/` contains markdown files for generated commands/skills/rules/agents.
    - (Manual check) Launch Hermes and verify it recognizes the generated environment.
